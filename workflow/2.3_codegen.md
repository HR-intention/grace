## Phase 6-7: Build & Test Loop

**This is a single loop. You cannot exit it until both the build command AND the smoke test (from Language config) pass. If you skip the smoke test for any reason, return FAILED, never SUCCESS.**

**DO NOT skip the smoke test. "Service not running" is NOT a valid reason to skip — you must start it yourself.**

### CRITICAL — Anti-loop safeguards (violation = immediate FAILED):

- **NEVER rerun the same smoke test or build command without changing code first.** If you did not edit a source file since the last attempt, you MUST NOT retry. Same code = same result. Always.
- **Maintain a fix log.** Before each retry, write down: (1) what error you saw, (2) what file you changed, (3) what you changed and why. If you cannot fill in all three, you are looping — stop and return FAILED.
- **Read server logs before every fix.** The smoke test error message alone is never enough to diagnose the problem. You MUST check the running service's stdout/stderr for the actual root cause (serialization error, missing field, wrong URL, panic, etc.) before touching any code.
- **3-strike rule on identical errors:** If you see the exact same error message 3 times, you have failed to fix it. Return FAILED immediately. Do not attempt a 4th time.
- **Maximum 7 total loop iterations** (build + test cycles combined). After 7, return FAILED regardless. This prevents runaway loops.

### The Loop:

```
1. Build
2. If build fails -> read error -> fix the specific code -> go to 1
3. Start service (if not running) -> load creds -> run smoke test
4. If smoke test fails -> read SERVER LOGS -> identify root cause -> fix the specific code -> go to 1
   (You MUST change code between step 4 and going back to step 1. No change = FAILED.)
5. If smoke test fails with credential error -> ask user -> go to 3
6. Both pass -> exit loop -> return SUCCESS
```

### Step 1 — Build:

Run the `<build command from Language config>`. For Rust this is `cargo build --package connector-integration 2>&1`. For Python this is the `mypy`+`pytest --collect-only` chain from the Language config (run from the Python repo root).

If it fails, fix the errors and rebuild. Keep iterating as long as errors are changing/reducing. Give up only after 5+ cycles with the same errors 3 times in a row AND the errors are outside your connector's files. Remember: you must make a code change between each retry — never rebuild without changing something.

If giving up on build: return FAILED with the build error details.

### Step 4 — Run Authorize with the appropriate payment method.

Use the `<smoke test command from Language config>`. For Rust this is a `grpcurl` call (template below); for Python this is the `pytest` integration test selecting the `test_authorize` case (the test harness builds the equivalent request body — populate fixtures from `creds.json`).

The Authorize request `payment_method` must use the type supported by the connector for this flow. Pick the appropriate method based on the connector's tech spec. Connector credentials/headers are **connector-specific** — populate them from `creds.json` (step 3). Only include credentials/headers that exist in the connector's credentials; do not guess or add unused ones.

**grpcurl template (Rust — common fields):**

```bash

  -H 'x-connector: {connector}' \
  -H 'x-auth: <AUTH_TYPE_FROM_CREDS>'  \

  # Example: -H 'x-api-key: <value>' -H 'x-key1: <value>' ...

    "request_ref_id": {"id": "test_{connector}_{flow}_001"},

    "minor_amount": <MINOR_AMOUNT>,

    "payment_method": {
      <PAYMENT_METHOD — see options below for BankDebit, adapt for other flows>
    },
    "email": {"value": "test@example.com"},
    "address": {
      "billing_address": {
        "first_name": {"value": "John"},
        "last_name": {"value": "Doe"},
        "line1": {"value": "123 Test St"},
        "city": {"value": "Test City"},
        "state": {"value": "CA"},
        "zip_code": {"value": "12345"},
```

**Fields to adapt per connector:**

### CRITICAL — You MUST validate the smoke test output. Running the smoke test is NOT the same as passing it.

After every smoke test invocation (grpcurl for Rust, pytest for Python), inspect the FULL output and determine PASS or FAIL:

**PASS (all must be true):**
- No line starts with `Error invoking method` or `Failed to`
- Response contains a valid JSON object with `"status"` field
- `status_code` in the response is 2xx (200-299). Any other code (400, 401, 500, etc.) is a FAIL.
- Status is one of: `authorized`, `PENDING`, `charged`, `REQUIRES_CUSTOMER_ACTION`
- No `error` or `errorMessage` field in the response (or error field is null/empty)

**FAIL (any one means test FAILED):**
- (Rust) Output contains `Error invoking method` -> grpcurl itself failed (wrong field name, wrong method, connection refused)
- (Python) pytest reports `FAILED` for `test_authorize`, an assertion error, an unhandled exception, or a non-zero exit code
- `status_code` is NOT 2xx (e.g., 400, 401, 403, 500) -> connector rejected the request
- Output contains `PAYMENT_FLOW_ERROR`, `INTERNAL`, `UNIMPLEMENTED`, `UNKNOWN` -> server error
- Response has `"status": "failed"` or `"status": "FAILURE"`
- Response has a non-null `"error"` object with a message
- No JSON response at all (empty output, timeout, crash)

**If the test FAILS, you are STILL inside the loop. You MUST diagnose, fix, rebuild, and retest. You CANNOT return SUCCESS. Returning SUCCESS after a failed test is a critical violation.**

**When the smoke test returns an error (including 500, PAYMENT_FLOW_ERROR, pytest failure, or any non-success response):**

1. **Check the server logs FIRST** — the backend service is running in the background. Check its stdout/stderr for the actual error:
   ```bash
   # Server logs are in the background process output. Check with:
   jobs -l                        # find the PID
   cat /proc/<PID>/fd/1 2>/dev/null || true   # or check recent terminal output
   ```
   The server logs will contain the real error (e.g., serialization failure, missing field, wrong URL, auth rejection). The smoke test response alone is often too vague to diagnose.

2. **Do NOT skip testing because you got an error.** An error means something needs to be fixed, not that testing should be abandoned. Diagnose -> fix -> rebuild -> retest.

### Error classification for step 4 failures:

The transformers module path differs per language: `transformers.rs` (Rust) or `transformers.py` (Python). The table below uses the Rust path as the example — substitute the Python path when `{LANG}` is `python`.

| Type | Signs | Action |
|---|---|---|
| Transport/config | Connection refused, wrong method name (Rust grpcurl); test client misconfigured / can't reach server (Python pytest) | Fix the smoke test invocation, check proto/HTTP route, retry from step 4 |
| Credentials | 401/403, "unauthorized" | Ask user for correct creds, retry from step 3 |
| Request format | 400/422, "missing field" | Check server logs, fix request struct in `transformers.{rs|py}` -> **go back to step 1 (rebuild)** |
| Response parsing | Deserialization error, panic, pydantic/marshalling error | Check server logs, fix response struct in `transformers.{rs|py}` -> **go back to step 1 (rebuild)** |
| Server error | 500, PAYMENT_FLOW_ERROR | **Check server logs for root cause**, fix connector code -> **go back to step 1 (rebuild)** |
| Code/logic | Wrong status, wrong amount, wrong header | Check server logs, fix connector code -> **go back to step 1 (rebuild)** |

**Never say "skip the full end-to-end test" or "the build passed so the implementation is correct". A passing build only proves syntax/types. The smoke test proves correctness. You must keep looping until the smoke test succeeds — but every loop iteration MUST include a real code change. If you find yourself about to rerun a test without having changed any code, STOP — you are in an infinite loop. Return FAILED immediately.**

Keep looping as long as errors are changing and you are making meaningful fixes. Return FAILED when:
- Same error appears 3 times (3-strike rule)
- 7 total loop iterations reached
- You cannot diagnose the root cause after reading server logs
- The error is outside your connector's files and you cannot fix it
