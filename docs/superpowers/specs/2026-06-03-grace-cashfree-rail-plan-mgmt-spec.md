## 1. Problem statement

Two lens releases hand-edited four Grace-generated files in the Cashfree subscriptions connector:

- **lens 0.3.0 — customer-chosen rail.** `CreateSubscriptionRequest.rail: MandateRail` became `rails: list[MandateRail] | None`; the connector now translates it into Cashfree's `authorization_details.payment_methods` allow-list, and surfaces the customer's **realized** rail back on both the auth webhook and `sync_subscription`.
- **lens 0.4.0 — plan upgrade/downgrade.** New `create_plan` (`POST /plans`) and `change_plan` (`CHANGE_PLAN` manage action) on `MandateConnector`.

Those edits live in `subscriptions/{connector,webhooks,status_map,models}.py`, every one bearing the `DO NOT EDIT — autogenerated by Grace` marker. Per **constitution §8**, hand-patches to marked files are not a durable mechanism — the next `grace generate` silently reverts them. Grace must be extended so regeneration *reproduces* the behavior.

**It cannot today.** Grace's pattern docs still encode a singular `rail`, have no realized-rail step, and no plan-management methods. More importantly — and this is the finding the source regen prompt understates — **Grace's pipeline *code* actively forbids the new shape**: the prompt's "`LOCKED … EXACT — no more, no less`" field guard would reject the three new webhook/sync fields, and the rubric's fixed per-domain method set would never gate `create_plan`/`change_plan`. Until Grace's **inputs *and* code** move to the new contract, every Cashfree regen reverts the hand-edits.

---

## 2. Goals

- **G1 — Reproduce.** `grace generate cashfree --domain subscriptions` reaches **behavioral parity** with the current hand-edited lens reference (rail + plan-mgmt), with **zero** hand-edits to marked files.
- **G2 — Generic, not special-cased.** The rulebook methodology generalizes to any PSP (offer-a-set-of-rails, realized-rail-on-auth-success, plan create/change). Cashfree specifics (wire keys, method strings, the American/British spelling split) stay in `connector_docs/cashfree.md`.
- **G3 — Pass the gates.** Regen output passes the §8 parity tests and the §9 quality gates: `mypy --strict` with **no `Any`**, `pytest --cov` **≥ 80 %**, `black -l 100`, the public-surface snapshot, and a well-formed §4 marker.
- **G4 — Durable.** The change survives future regenerations — this is the defining requirement, not a nice-to-have. Anything that only works because a file was hand-touched is a failure.

---

## 3. Non-goals

- **On-demand subscription create mode + the Charge Mandate / controlled-execution API** (on-demand + UPI-only). Deferred; explicitly **not** generated. (Source §4e "Out of scope".)
- **Re-shaping any locked lens type/enum.** The 0.3.0/0.4.0 domain contract is shipped and §8-locked; Grace consumes it from `lens.domain_types`/`lens.enums` and must never redefine, rename, or re-shape it.
- **Touching `orders/` or `core/`.** Incremental subscriptions-only regen (SUBPROJECT_GRACE_CODEGEN §3.2).
- **Changing the `max_mandate_amount(rail: MandateRail)` introspection signature.** It keeps its single `rail` param — it is the introspection method, unrelated to the request-field `rail → rails` rename. (Easy to conflate; do not.)
- **Committing lens-side `connector_docs` or these governance docs.** The user owns those commits.

## 5. The reframing — this is NOT a pure rulebook/doc change

The source prompt's §0/§7 say the durable fix is "the rulebook + `connector_docs`." That is **necessary but not sufficient.** **Five** Grace *pipeline-code / registration / mirror-doc* surfaces must also move, or the regen cannot reproduce the reference:

1. **`src/grace/pipeline/prompt.py` — the "EXACT — no more, no less" locked-field guard** (✓ verified):
   - `prompt.py:514-518` and `:602-606` declare `MandateWebhookEvent` locked to exactly `event_type, psp_mandate_ref, psp_event_id, occurred_at, mandate_status, debit, raw`. The backend is told emitting any other field is forbidden — so `realized_rail`/`authorization_reference`/`payment_group` **cannot** be generated until this list is extended.
   - `prompt.py:512` and `:561` lock `SyncSubscriptionResponse` to `status, next_charge_at, last_debit, raw` — same problem for the sync path.
2. **`src/grace/quality_rubric.py:452-462` — `_V2_DOMAIN_METHODS["subscriptions"]`** (✓ verified): a fixed 9-method list (5 lifecycle + 4 introspection). It does not *reject* extra methods, but it will never *require/score* `create_plan`/`change_plan` — so the gate can't enforce them. Must be extended.
3. **Locked-type mirror docs** `rulesbook/codegen/python/domain_types.md` (✓ `rail` at L118; `MandateWebhookEvent` L177-185; `SyncSubscriptionResponse` L145-150) and `connector_abc.md` (○ the `MandateConnector` ABC lists 5 lifecycle + 4 introspection, no plan methods). These mirror the lens contract for the backend; if they disagree with the shipped lens types the backend gets contradictory guidance.

4. **`src/grace/pipeline/context.py:30-43` — the hardcoded rulebook file list** (✓ verified): `DOMAIN_PATTERN_FILES["subscriptions"]` enumerates exactly four pattern files, and `default_rulebook_paths` loads *only* listed files (raising if one is missing). A new `pattern_create_plan.md` dropped on disk is **never read by the backend** unless added here. (Alternative: fold the `create_plan` pattern into an already-registered file — but the registration chokepoint must be addressed either way.)
5. **`src/grace/pipeline/prompt.py:83-96` — `_IMPORTS_SUBSCRIPTIONS`** (and `_IMPORTS_ALL` `:98-112`) (✓ verified): imports only the non-plan domain types — **no** `CreatePlanRequest`/`CreatePlanResponse`/`ChangePlanRequest`. A generated `create_plan`/`change_plan` referencing them fails `mypy --strict` (name-undefined) → the G3 gate fails. The mirror docs `domain_types.md` + `connector_abc.md` omit them too. All three surfaces must add the plan types.

What does **not** need a code change (✓/○ verified): `compose.py` (the merged facade is structural — `class <Psp>Connector(<Psp>Orders, <Psp>Subscriptions)` inherits new methods automatically) and `domain_breakdown.py` (path/keyword bucketing — see [P1-1](#p1)). And the CREATE-side `rails` field does **not** need a `prompt.py` edit — the request field flows from `domain_types.md` + the real lens type (prompt.py's subscriptions block defers field names to `domain_types.md`); only the *webhook/sync response* fields are hard-listed in `prompt.py`.

## 6. Requirements

### P0 — Must-have (regen can't reach parity without these)

#### Capability A — Customer-chosen rail (CREATE)

- **A1.** Grace emits `create_subscription` that translates `request.rails` into the PSP allow-list:
  - **A1.1** Non-empty → **deduped, order-preserving union** of each rail's method list, iterating `request.rails` in given order.
  - **A1.2** `None` or empty → **omit** the allow-list field (serialize via `model_dump(exclude_none=True)`; set the field to `None`). **Never send `[]`.**
  - **A1.3** Any rail ∉ `supported_mandate_rails()` → raise `ConnectorError(NOT_SUPPORTED)` **before any HTTP call**.
  - *Lands in:* `pattern_create_subscription.md` (singular `rail` → plural `rails` + the union/omit/reject methodology, replacing the `_rail_to_payment_methods(request.rail)` scalar pattern at L80/L168); `domain_types.md:118` (`rail` → `rails`). Cashfree's per-rail method strings stay in `connector_docs` (done).
  - **Acceptance:** the five `test_create_subscription.py` rails cases (union → `["upi","enach","pnach","card"]`; single card → `["enach","pnach","card"]`; `None` → field absent; `[]` → field absent; unsupported → `NOT_SUPPORTED` with **no** HTTP call).

#### Capability B — Realized rail (READ, both paths)

- **B1.** Grace **permits and instructs** the three new fields on `MandateWebhookEvent` **and** `SyncSubscriptionResponse`.
  - *Lands in:* `prompt.py` locked-field blocks (`:512/:514-518` and `:561/:602-606`) — add the three fields to both lists; `domain_types.md` (L145-150, L177-185).
  - **Acceptance:** generated models include the fields; `mypy --strict` clean; public-surface snapshot matches.
- **B2.** Grace emits **webhook** realized-rail extraction: read the PSP's **American** `authorization_details`, gate on `payment_type=="AUTH" and payment_status=="SUCCESS"`, **null all three on failure even when the group is present**, coerce dict reads with `isinstance` (no `Any` leak), and parse **no** rail from the card-expiry/reminder event.
  - *Lands in:* `pattern_mandate_webhook.md` (new "realized rail (auth-success only)" step). Cashfree wire keys in `connector_docs`.
  - **Acceptance:** `test_webhook_router.py` — AUTH+SUCCESS+`upi` → `UPI_AUTOPAY`+ref+raw; `enach` → `CARD_EMANDATE`; **AUTH+FAILED with `payment_group` present → all three `None`** (the gotcha); expiry-reminder → `None`; AUTH-success with no `authorization_details` → all three `None`.
- **B3.** Grace emits **sync** realized-rail extraction: read the **British** `authorisation_details` typed field, extract only when `payment_group` is present; `last_debit` stays `None`.
  - *Lands in:* `pattern_sync_subscription.md` (realized-rail population + an explicit note that the wire key may differ from the webhook's — call out the **American-vs-British spelling split** as a per-PSP possibility, and that sync is the reconcile-backstop recovery path).
  - **Acceptance:** `test_sync_subscription.py` — British key + `upi`+ref+ACTIVE → `UPI_AUTOPAY`; `enach` → `CARD_EMANDATE` raw kept; null/absent → all `None`.
- **B4.** Grace emits a `map_payment_group_to_rail` mapper: **many→few** (`upi → UPI_AUTOPAY`; `enach/pnach/card/debit_card → CARD_EMANDATE`; absent/unknown → `None`), **case-insensitive**, **warn** on unknown non-empty, and **preserve the raw `payment_group`** on the event regardless of whether it mapped.
  - *Lands in:* `status_mapping.md` — add a **raw-preserving many→few** example. (Note: the current doc's mappers normalize-and-discard the source term; preserving the raw value alongside the normalized one is a methodology *shift*, not just an addition.)
  - **Acceptance:** the parametrized `test_map_payment_group_to_rail` (incl. case-insensitive `"UPI"`, and `unknown`/`None`/`""` → `None`).
